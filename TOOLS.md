# TOOLS.md - Local Notes

## 📓 NotebookLM 활용 규칙 (기본 행동 규칙)

### 일일 워크플로우

#### 🌅 매일 저녁 (작업 종료 시)
1. 당일 `memory/YYYY-MM-DD.md` 작성 완료 후
2. NotebookLM에 당일 파일 업로드: `nlm source add <NOTEBOOK_ID> --file <파일경로>`
3. NotebookLM에 일일 분석 질의:
   - "오늘 작업 로그를 VF/키/시스템으로 분류해줘"
   - "미해결/대기 중인 작업을 리스트업해줘"
   - "에러 발생 건의 원인과 해결책을 요약해줘"
4. NotebookLM 응답을 옵시디언에 저장 → Git 푸시

#### 📅 매주 월요일 (주간 리뷰)
1. 지난주 일일 파일 7개 NotebookLM에 업로드
2. 주간 성과 분석 질의:
   - "지난주 VF/키 프로젝트 진행률과 병목 원인 분석"
   - "이번 주 최우선 해결 과제 제안"
3. 결과를 `weekly_YYYYMMDD.md`로 저장

#### 📆 매월 말 (월간 정리)
1. 옵시디언 전체 파일 NotebookLM에 업로드
2. 중복/충돌 검증 질의:
   - "마스터 데이터 간 충돌/중복 내용 교차 검증"
   - "병합해야 할 파일 목록 제안"
3. 정리 후 Git 푸시

### 핵심 프롬프트 템플릿
```
# 분류
"업로드한 파일들을 VF 프로젝트/키 프로젝트/시스템/일일기록으로 분류해줘"

# 정리  
"오늘 논의된 내용을 마스터 데이터 규칙에 맞게 정리해줘"

# 요약
"오늘 발생한 에러의 증상, 원인, 해결책을 요약해줘"

# 검색
"최근 N일간 미해결 상태인 작업을 찾아줘"
```

### 고급 프롬프트 템플릿

#### 압축 프롬프트 (Raw → Compiled Truth)
```
"제공된 소스들을 분석해서 다음 구조로 재작성해줘:
## 현재 상태 (Compiled Truth)
현재까지 알려진 가장 정확한 정보
---
## 변경 이력 (Timeline)
- 날짜: 내용"
```

#### 검증 프롬프트 (모순/오래된 정보 탐지)
```
"업로드된 소스 간에 충돌하는 사실이나, 갱신되지 않은 과거 데이터를 찾아줘.
형식: [파일명] vs [파일명] — 모순 내용 — 어느 것이 최신인지"
```

#### 일일 분석 프롬프트
```
"오늘 작업 로그를 분석해서:
1. VF/키/시스템으로 분류
2. 미해결 작업 리스트
3. 내일 우선 작업 제안
4. Compiled Truth 갱신 필요 항목"
```

#### 주간 헬스체크 프롬프트
```
"지난주 데이터를 기반으로:
1. 모순/충돌하는 정보
2. 고아 페이지 (링크 없음)
3. 갱신 필요한 오래된 정보
4. 새로 만들어야 할 페이지"
```
